 STAGE 1: DISCOVER SCANS THE EXISTING PROJECT
 =============================================
 (runs once via /discover -- builds the project's knowledge base)

  /discover
      |
      v
  Existing codebase detected (has source files, package.json, etc.)
      |
      v
  4 ANALYSIS AGENTS LAUNCH IN PARALLEL:
  +------------------------------------------------------------------+
  |                                                                  |
  |  D1: Architecture Analyzer                                       |
  |  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~                                    |
  |  scans: package.json, tsconfig, imports, folder structure        |
  |  extracts:                                                       |
  |    - tech stack (languages, frameworks, versions)                |
  |    - module boundaries (src/, lib/, hooks/, agents/)             |
  |    - dependency graph (internal + external)                      |
  |    - integration points (APIs, databases, message queues)        |
  |    - architecture patterns (hooks-based, supervisor tree, etc.)  |
  |                                                                  |
  |  D2: Test Evaluator                                              |
  |  ~~~~~~~~~~~~~~~~~~~                                             |
  |  scans: test files, config (vitest, jest, mocha), CI pipelines   |
  |  extracts:                                                       |
  |    - test count by type (unit, integration, e2e)                 |
  |    - coverage percentage and thresholds                          |
  |    - test frameworks in use                                      |
  |    - gaps (untested modules, missing integration tests)          |
  |    - test naming conventions and patterns                        |
  |                                                                  |
  |  D5: Data Model Analyzer                                         |
  |  ~~~~~~~~~~~~~~~~~~~~~~~~                                        |
  |  scans: schemas, migrations, models, type definitions            |
  |  extracts:                                                       |
  |    - entity relationships                                        |
  |    - data store types (SQL, NoSQL, file-based)                   |
  |    - schema validation patterns                                  |
  |    - state management approach                                   |
  |                                                                  |
  |  D6: Feature Mapper + Behavior Extractor                         |
  |  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~                        |
  |  scans: routes, handlers, components, business logic             |
  |  extracts:                                                       |
  |    - API endpoints and their contracts                           |
  |    - UI pages/components and their props                         |
  |    - business domains and bounded contexts                       |
  |    - observable behavior as Given-When-Then AC                   |
  |    - existing error handling patterns                            |
  |                                                                  |
  +------------------------------------------------------------------+
      |
      |  all 4 agents complete, results merged
      |
      v
  OPTIONAL DEEP ANALYSIS (standard+ depth):
  +------------------------------------------------------------------+
  |                                                                  |
  |  D16: Security Reviewer (LLM-based code review)                 |
  |  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~                 |
  |  Claude reads code and checks for:                               |
  |    - hardcoded secrets, API keys, passwords in source            |
  |    - raw SQL / unsanitized input patterns                        |
  |    - weak auth (MD5, missing JWT expiry, no bcrypt)              |
  |    - missing input validation at boundaries                      |
  |    - findings categorized against OWASP Top 10                   |
  |  NOTE: this is LLM code review, not automated SAST/DAST.        |
  |        no external tools (npm audit, Snyk, ZAP) are invoked.    |
  |        see ROADMAP at the bottom of this file.                   |
  |                                                                  |
  |  D17: Tech Debt Analyzer  --> code smells, outdated deps, TODOs |
  |  D18: Performance Profiler --> bottlenecks, N+1 queries, caching|
  |  D19: Ops Readiness       --> logging, monitoring, health checks|
  |                                                                  |
  +------------------------------------------------------------------+
      |
      v
  D3: CONSTITUTION GENERATOR
  ~~~~~~~~~~~~~~~~~~~~~~~~~~~
  reads all analysis results + project type
  generates governance articles tailored to THIS project:

    Article I:   Specification Primacy
    Article II:  Test-First Development
    Article III: Security by Design
    Article IV:  Explicit Over Implicit
    Article V:   Simplicity First
    Article VI:  Code Review Required
    Article VII: Artifact Traceability
    ...up to 14 articles

  thresholds calibrated to actual project:
    - coverage gate set to current baseline (e.g., 82%), not arbitrary
    - test types required based on what D2 found exists
    - security articles tuned to project's auth patterns
      |
      v
  DISCOVERY OUTPUTS (committed to git):
  +------------------------------------------------------------------+
  |                                                                  |
  |  docs/isdlc/constitution.md                                      |
  |    --> 14 governance articles, project-specific thresholds       |
  |    --> enforced by hooks on EVERY phase, EVERY gate              |
  |                                                                  |
  |  docs/project-discovery-report.md                                |
  |    --> unified analysis: tech stack, architecture, patterns      |
  |    --> baseline for all future design decisions                  |
  |                                                                  |
  |  docs/isdlc/test-evaluation-report.md                            |
  |    --> coverage by type, gaps, framework details                 |
  |    --> read by test strategy + implementation agents             |
  |                                                                  |
  |  docs/requirements/reverse-engineered/                           |
  |    --> Given-When-Then AC extracted from existing code           |
  |    --> behavior captured BEFORE any changes are made             |
  |                                                                  |
  |  tests/characterization/                                         |
  |    --> test.skip() scaffolds documenting current behavior        |
  |    --> regression safety net for future changes                  |
  |                                                                  |
  +------------------------------------------------------------------+
      |
      |  These files now exist on disk, in git.
      |  They are the project's knowledge base.
      |
      v
  DISCOVERY COMPLETE. Project is ready for workflows.

 STAGE 4: KNOWLEDGE IS KEPT FRESH
 =================================
 (3 mechanisms prevent stale knowledge from causing bad decisions)


  FRESHNESS 1: CODEBASE HASH (detects drift between add and build)
  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  /isdlc add "feature"
      |
      v
  meta.json: { "codebase_hash": "a3f8b21" }    [git rev-parse --short HEAD]
      |
      |  ... time passes, team pushes code ...
      |
      v
  /isdlc analyze "REQ-0042"   or   /isdlc build "REQ-0042"
      |
      v
  Compare: meta.codebase_hash ("a3f8b21") vs HEAD ("f92c1d4")
      |
      +-- match?  --> "Analysis current. Proceed."
      +-- differ? --> "Codebase changed since analysis ({N} commits)."
                      [R] Re-analyze  |  [C] Cancel


  FRESHNESS 2: SESSION CACHE HASH (detects framework/source changes)
  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  rebuildSessionCache() hashes 252 source files  [common.cjs:4321]
      |
      v
  <!-- SESSION CACHE: Hash: 0f56e68d -->
      |
      |  If ANY source file changes, hash changes.
      |  Cache is rebuilt on next session start.
      |
      v
  Every conversation gets FRESH project knowledge.


  FRESHNESS 3: CONSTITUTION VALIDATION (enforced every phase exit)
  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  Agent tries to declare phase complete
      |
      v
  constitution-validator.cjs reads:
    docs/isdlc/constitution.md       (the CURRENT file on disk)
    state.phases[phase].constitutional_validation
      |
      +-- Articles not checked --> BLOCKED (must read and comply)
      +-- All compliant        --> PASS (advance to next phase)
      |
      v
  Constitution is ALWAYS read fresh from disk at gate time.
  Even if the constitution was updated mid-workflow,
  the next gate check picks up the changes.



 THE FULL CHAIN (end to end):
 ============================

  /discover (once)
      |
      v
  constitution.md + discovery-report.md + test-evaluation.md
  (project knowledge captured from actual codebase)
      |
      v
  rebuildSessionCache()
  (knowledge packed into session-cache.md with HTML delimiters)
      |
      v
  inject-session-cache.cjs --> stdout --> LLM CONTEXT WINDOW
  (8 lines of code. knowledge enters every conversation.)
      |
      v
  Orchestrator extracts <!-- SECTION: DISCOVERY_CONTEXT -->
  (pulls project knowledge from session context)
      |
      v
  Delegation prompt to Phase 01 (with DISCOVERY CONTEXT block)
  ("Use discovery as baseline. Extend, don't redesign.")
      |
      v
  Phase 01 produces requirements-spec.md (reads discovery + user input)
      |
      v
  state.json updated (phase 01 complete)
  phase-loop-controller allows Phase 02
      |
      v
  Phase 02 reads requirements-spec.md + discovery context
  (explicit file read, not memory)
      |
      v
  Phase 02 produces architecture-overview.md
      |
      v
  constitution-validator checks compliance (reads constitution from disk)
      |
      v
  Phase 03 reads architecture-overview.md + discovery context
      |
      v
  ... each phase reads previous phase's files ...
      |
      v
  Phase 08 reads ALL prior artifacts for final review
      |
      v
  Merge to main. Workflow archived in workflow_history.
  Roundtable memory updated. Codebase hash refreshed.
