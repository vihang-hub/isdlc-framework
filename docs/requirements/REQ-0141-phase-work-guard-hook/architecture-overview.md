# Architecture Overview: Execution Contract System

**Slug**: REQ-0141-phase-work-guard-hook
**Version**: 1.0.0

---

## 1. Architecture Options

### Option A: Core-First Evaluator with Provider Adapters

| Aspect | Detail |
|---|---|
| **Summary** | Contract evaluation logic lives in `src/core/validators/` (provider-neutral). Claude and Codex each have thin adapter layers that call the core evaluator at their respective enforcement points |
| **Pros** | Single evaluation codebase, cross-provider parity guaranteed, testable in isolation, follows existing core/ pattern |
| **Cons** | Requires core/ to have access to state helpers currently in `.claude/hooks/lib/common.cjs` |
| **Pattern Alignment** | Matches existing `src/core/validators/validate-phase.js` pattern used by Codex |
| **Verdict** | **Selected** |

### Option B: Dual Evaluators (Hook + Core)

| Aspect | Detail |
|---|---|
| **Summary** | Claude gets a hook-based evaluator in `src/claude/hooks/`, Codex gets a core evaluator in `src/core/validators/`. Both read the same contract schema but are separate implementations |
| **Pros** | No dependency bridging needed, each provider owns its enforcement |
| **Cons** | Two codebases to maintain, drift risk, violates DRY, testing burden doubles |
| **Pattern Alignment** | Against the established core/ consolidation pattern from Codex integration |
| **Verdict** | Eliminated — drift risk too high for a system that must guarantee parity |

### Option C: Hook-Only with Codex Shim

| Aspect | Detail |
|---|---|
| **Summary** | Evaluator lives entirely in `.claude/hooks/`. Codex calls into it via a bridge/shim layer |
| **Pros** | Minimal new files, reuses existing hook infrastructure |
| **Cons** | CJS/ESM boundary issues, hooks are Claude-specific by design, Codex becomes a second-class consumer |
| **Pattern Alignment** | Against the provider-neutral core/ architecture established in Codex integration |
| **Verdict** | Eliminated — wrong direction for cross-provider parity |

## 2. Selected Architecture (ADRs)

### ADR-001: Core-First Evaluation with Provider Adapters

- **Status**: Accepted
- **Context**: The execution contract system must work identically across Claude (hook-based) and Codex (checkpoint-based) providers. The Codex integration established `src/core/` as the provider-neutral layer with `src/providers/{claude,codex}/` as thin adapters.
- **Decision**: Contract schema, evaluation logic, reference resolution, and violation reporting live in `src/core/validators/`. Claude calls the evaluator from the phase-loop controller (isdlc.md STEP 3e). Codex calls it from `validatePhaseGate()` in `runtime.js`. State helpers (`writeContractViolation`, `readContractViolations`, `clearContractViolations`) live in `.claude/hooks/lib/common.cjs` for Claude and are bridged to core for Codex via the existing `src/core/bridge/validators.cjs` pattern.
- **Rationale**: Single codebase eliminates drift. Core/ is already the established neutral ground. Provider adapters are thin call-throughs.
- **Consequences**: Core evaluator must not depend on Claude-specific hook context (stdin JSON, tool interception). It receives a provider-neutral input: `{ state, contractEntry, projectRoot }`. Provider adapters MUST remain thin call-throughs that only translate provider-specific context into the neutral input shape. If Claude and Codex start shaping inputs differently, parity will drift despite the shared core — adapter thickness is a code review signal.

### ADR-002: Pre-Generated Contracts with Config Override

- **Status**: Accepted
- **Context**: Contracts must be deterministic and not computed at runtime. User configuration must be honored.
- **Decision**: Built-in contracts ship in `.claude/hooks/config/contracts/`. User overrides go in `.isdlc/config/contracts/`. Override resolution: `.isdlc/config/` wins for the same `execution_unit` + `context` key. Contracts are generated by `bin/generate-contracts.js` which reads from `PHASE_AGENT_MAP`, `artifact-paths.json`, `skills-manifest.json`, `external-skills-manifest.json`, `roundtable.yaml`, and `workflows.json`.
- **Rationale**: Follows the existing source/installed override pattern used by other config files. Pre-generation means zero runtime computation cost.
- **Consequences**: Config changes require re-running the generator. Staleness detection (FR-006) mitigates the risk of running against outdated contracts.

### ADR-003: Additive Enforcement Layer

- **Status**: Accepted
- **Context**: The framework has 28+ hooks that individually enforce narrow concerns. Replacing them is risky and unnecessary.
- **Decision**: The contract evaluator is additive — it does NOT replace existing hooks (gate-blocker, blast-radius-validator, etc.). Existing hooks continue to run unchanged. The contract evaluator catches what falls through the cracks: agent engagement, skill usage, UX compliance, cleanup activities.
- **Rationale**: Zero disruption to current enforcement. Incremental adoption. Over time, individual hooks could be migrated to contract entries, but that is a separate effort (out of scope).
- **Consequences**: Some enforcement is duplicated (e.g., artifact presence checked by both gate-logic and contract evaluator). This is intentional — defense in depth.

### ADR-004: In-Memory Mutator Pattern for State Helpers

- **Status**: Accepted
- **Context**: State helpers in `common.cjs` use two patterns: some read/write disk directly, some mutate passed-in state objects. Newer helpers use the in-memory pattern.
- **Decision**: `writeContractViolation(state, entry)`, `readContractViolations(state)`, and `clearContractViolations(state)` are pure in-memory mutators. The caller is responsible for persisting state to disk. The core evaluator returns a structured result `{ violations, warnings, stale_contract }` — the caller owns the single persistence point.
- **Rationale**: Consistent with newer helper pattern. Easier to test (no disk I/O in unit tests). Caller controls write timing (can batch multiple mutations). Returning a structured result instead of mutating state inside the evaluator keeps the evaluator pure and avoids mixed mutator/write patterns.
- **Consequences**: Callers must explicitly persist after mutation. The evaluator never writes state — it only returns what the caller should write. This is documented in JSDoc.

### ADR-005: Generalized Reference Resolution

- **Status**: Accepted
- **Context**: Contract entries need to reference dynamic data (artifact paths, skill lists) without hardcoding values that drift.
- **Decision**: A single `resolveRef(ref, configs)` function handles all `$ref` objects in contracts. Reference format: `{ "$ref": "<source>", ...params }`. Initial sources: `artifact-paths` (resolves to file paths per phase), `skills-manifest` (resolves to skill IDs per agent). New sources are added by registering a resolver function.
- **Rationale**: One resolution model prevents ad-hoc token parsing. Extensible for future reference types.
- **Consequences**: The evaluator must load referenced config files. These are cached on first load per evaluation cycle (not per reference).

### ADR-006: PHASE_AGENT_MAP as Stable Export

- **Status**: Accepted
- **Context**: The contract generator needs the phase-to-agent mapping. `PHASE_AGENT_MAP` exists in `common.cjs:2503` but may not be an explicitly exported stable symbol.
- **Decision**: Verify `PHASE_AGENT_MAP` is in the `module.exports` of `common.cjs`. If not, add it as an intentional, documented export. Add a test that guards the export exists and contains all expected phase keys — this prevents accidental removal or renaming. The generator imports it directly.
- **Rationale**: No duplicate mapping. Single source of truth for phase-to-agent relationships. A guarding test makes the export contract explicit rather than coupling to an incidental internal symbol.
- **Consequences**: Changes to `PHASE_AGENT_MAP` affect contract generation. This is intentional — the map IS the authority. The guarding test will fail if someone removes a phase mapping, making the dependency visible.

### ADR-007: Override Conflict Rule — Full Replacement, Not Deep Merge

- **Status**: Accepted
- **Context**: When a user provides an override contract in `.isdlc/config/contracts/` for the same `execution_unit` + `context`, the system needs a deterministic conflict rule.
- **Decision**: Override is **full replacement** at the contract entry level. If the user provides a contract for `execution_unit: "06-implementation"` + `context: "feature:standard"`, it completely replaces the shipped default for that key. No deep merge of individual fields.
- **Rationale**: Deep merge creates ambiguity — if the user overrides `violation_response.skills_missing` but not `violation_response.artifacts_missing`, does the shipped default for `artifacts_missing` survive? Full replacement is unambiguous: the user's file is the complete truth for that entry. If they want to keep a shipped default, they copy it into their override and modify what they need.
- **Consequences**: User overrides must be complete contract entries. Partial overrides (e.g., just changing one violation response) require copying the full entry. This is a trade-off for determinism.

### ADR-008: Codex Instruction Injection is Advisory-Only

- **Status**: Accepted
- **Context**: Codex `projectInstructions()` can inject contract expectations into the agent's instruction bundle so agents know upfront what's expected. However, this creates two truth surfaces: the injected instructions and the runtime evaluation.
- **Decision**: Contract injection into `projectInstructions()` is **advisory-only** — it helps agents produce compliant output but is NOT enforcement. Runtime evaluation via `validatePhaseGate()` is the sole authority for contract compliance. The injected text is informational ("This phase expects artifacts X, Y, Z") not imperative ("You MUST produce X, Y, Z or you will be blocked").
- **Rationale**: If injection were treated as enforcement, the framework would have two truth surfaces: what the agent was told and what the evaluator checks. When they diverge (and they will — instruction text is lossy), the system becomes non-deterministic. One authority (runtime evaluator) eliminates this.
- **Consequences**: Agents may still produce non-compliant output despite advisory injection. The runtime evaluator catches this. This is the intended design — advisory reduces violations, enforcement catches what leaks through.

## 3. Technology Decisions

| Technology | Version | Rationale | Alternatives Considered |
|---|---|---|---|
| JSON contract files | N/A | Matches existing config format (artifact-paths.json, iteration-requirements.json). Parseable by both CJS and ESM. | YAML (rejected — no YAML parser in hooks, CJS boundary), JS modules (rejected — not human-readable config) |
| `src/core/validators/` location | N/A | Established provider-neutral validator location from Codex integration | `.claude/hooks/` (rejected — Claude-specific), new `src/contracts/` (rejected — unnecessary new directory) |
| `bin/generate-contracts.js` (ESM) | N/A | Follows existing `bin/` CLI tool pattern (isdlc.js, rebuild-cache.js) | CJS script in `src/antigravity/` (rejected — generator is a user-facing CLI tool, not an internal script) |

## 4. Integration Architecture

### Integration Points

| ID | Source | Target | Interface | Data Format | Error Handling |
|---|---|---|---|---|---|
| INT-001 | Phase-loop controller (isdlc.md STEP 3e) | Core evaluator | Function call via bridge | `{ state, contractEntry, projectRoot }` → `{ violations[] }` | Fail-open: empty violations on error |
| INT-002 | Codex `validatePhaseGate()` | Core evaluator | Direct import | Same as INT-001 | Fail-open: same |
| INT-003 | Core evaluator | `artifact-paths.json` | File read (cached) | JSON | Missing file → skip artifact checks |
| INT-004 | Core evaluator | `skills-manifest.json` | File read (cached) | JSON | Missing file → skip skill checks |
| INT-005 | Core evaluator | `state.json` | Passed-in state object | In-memory JSON | Missing fields → skip assertions |
| INT-006 | Generator | `PHASE_AGENT_MAP` | CJS require | JS object | Missing export → generator fails with clear error |
| INT-007 | Generator | Config files (7 sources) | File reads | JSON/YAML/MD | Missing optional files → skip that source, warn |
| INT-008 | Codex `projectInstructions()` | Contract file | File read | JSON | Missing contract → no injection (fail-open) |
| INT-009 | Contract loader | `.isdlc/config/contracts/` | File read (override) | JSON | Missing override → use shipped default |

### Data Flow

```
Config change
  → User runs `bin/generate-contracts.js`
  → Generator reads: PHASE_AGENT_MAP, artifact-paths.json, skills-manifest.json,
    external-skills-manifest.json, roundtable.yaml, workflows.json, agent .md files
  → Generator writes: .claude/hooks/config/contracts/*.json (shipped)
  → User overrides: .isdlc/config/contracts/*.json

Runtime (Claude):
  Phase executes → isdlc.md STEP 3e → load contract for {execution_unit, context}
  → resolve $refs → check agent (skill_usage_log) → check artifacts (disk)
  → check state assertions (state.json) → check presentation (output shape)
  → violations[] → writeContractViolation(state, entry) → persist state
  → phase-loop reads contract_violations → dispatch remediation

Runtime (Codex):
  Phase executes → validatePhaseGate() → load contract → same evaluation
  → violations[] → write to state → phase-loop.js reads → remediation

Runtime (non-workflow):
  analyze/discover/add completes → inline evaluation at handler end
  → same contract loading and evaluation → violations logged
```

### Synchronization
- Contract files are static (no concurrent write risk)
- `contract_violations[]` in state.json follows the same append-with-dedup pattern as `pending_escalations` — no concurrent modification concern (single-agent execution per phase)
- Config file reads during evaluation are cached per evaluation cycle — no repeated I/O
