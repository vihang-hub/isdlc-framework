## 2. Entry Points

### Recommended Entry Point: `workflow-retry.cjs` (REQ-0051)

**Rationale**: Retry is simpler (no V8 index regression), self-contained, and REQ-0052 depends on it. Building retry first establishes the `recovery_action` pattern that rollback extends.

### Implementation Chain
1. `workflow-retry.cjs` — new script, reads/writes state.json
2. `state-logic.cjs` — add `recovery_action` exception to V8 checks
3. `workflow-rollback.cjs` — new script, uses same recovery_action pattern
4. `ANTIGRAVITY.md.template` — add intent detection entries
5. Tests — cover both scripts and V8 exception

## 4. Risk Zones

| ID | Risk | Area | Likelihood | Impact | Mitigation |
|----|------|------|-----------|--------|------------|
| R1 | V8 exception too broad | `state-logic.cjs` | Low | High | Scope `recovery_action` to specific type + phase; clear after recovery completes |
| R2 | Cross-location inconsistency after rollback | `state.json` | Medium | Medium | Rollback script updates BOTH `active_workflow.phase_status` and `phases[].status` atomically |
| R3 | Stale gate_validation after retry | `state.json` phases | Medium | Low | Retry script clears `gate_validation` for the reset phase |
| R4 | Intent detection ambiguity ("start over") | ANTIGRAVITY.md | Medium | Low | Disambiguate: ask user "retry current phase or go back to an earlier one?" |
| R5 | Existing `supervised_review` exception becomes redundant | `state-logic.cjs` | Low | Low | Keep backward compat; `recovery_action` is additive, not replacing `supervised_review` |

## 5. Summary

**Overall scope**: Medium (6-8 files, 2 new scripts, 1 governance modification)
**Overall risk**: Low-Medium (V8 exception is the only area requiring careful implementation)
**Recommendation**: Proceed. Build retry first (REQ-0051), then rollback (REQ-0052). The shared `recovery_action` pattern keeps V8 changes minimal and extensible.

### Key Decisions
| Decision | Rationale |
|----------|-----------|
| Artifacts preserved on disk for both retry and rollback | Simplest approach; agents overwrite naturally; avoids destructive file operations |
| No retry count cap | User is in control; tracking for observability only |
| `recovery_action` flag generalizes V8 exception | Extensible pattern — avoids special-casing each recovery type |
| Rollback requires confirmation; retry does not | Rollback is destructive to phase state (multiple phases reset); retry is low-risk (same phase) |
| Rollback target must be in workflow's phase array | Prevents invalid state from light-mode skipped phases |
